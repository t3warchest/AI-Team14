Abstract
Recent advances in large language models have enabled agentic systems capable of performing structured data analysis with minimal human intervention. This paper presents a comparative study of a four-agent orchestration system — comprising Planner, ResearchAgent, DataScientist, and CodeReviewerAgent roles built on AutoGen — against a single-agent oneshot baseline, evaluated across fifty analytical tasks spanning five real-world Kaggle datasets covering retail, healthcare, finance, education, and mobility domains. We introduce a structured evaluation framework spanning output correctness, process quality, latency, robustness, and an LLM-as-judge scoring module for subjective analytical tasks. Across 10 representative benchmark tasks, the multi-agent pipeline achieves higher correctness on complex reasoning tasks at the cost of 3–6× higher latency. The single-agent baseline wins on simple factual queries.
01

Introduction

Large language models have significantly expanded the range of tasks that can be automated through natural language interaction, including code generation, structured reasoning, and tool-assisted execution. One particularly promising application is end-to-end data analytics, where LLM-based systems can translate analytical requests into executable workflows involving data loading, cleaning, aggregation, visualization, and interpretation.

A single monolithic LLM call — the "oneshot" approach — often fails silently: it may propagate incorrect assumptions, generate misleading visualizations, or produce plausible but logically flawed conclusions without explicit failure signals. Orchestrating multiple specialized agents introduces an explicit planning-review loop that catches these errors before they reach the user.

Core Question

Does decomposing the analytics workflow into specialized collaborating agents produce measurably better outputs than a strong single-agent baseline — and for which task types is that investment worthwhile?

Contributions

  • A multi-agent data analytics framework (Planner → ResearchAgent → DataScientist → CodeReviewerAgent → Result Summarizer) built on Microsoft AutoGen.
  • A single-agent baseline using the same underlying GPT-OSS 20B model and execution environment for controlled comparison.
  • A benchmark suite of 50 analytical tasks across five real-world Kaggle datasets spanning retail, healthcare, finance, education, and mobility domains.
  • Empirical results showing multi-agent orchestration wins 8/10 representative tasks, with the largest gains on complex multi-step reasoning tasks.
  • An ablation study confirming the CodeReviewerAgent as the single highest-value component of the multi-agent pipeline.
02

Related Work

Our work sits at the intersection of three active research threads: LLM code generation, multi-agent coordination, and LLM-based evaluation.

AutoGen (Wu et al., 2023)
Conversational multi-agent framework. Our work builds directly on AutoGen using SelectorGroupChat for the multi-agent pipeline and RoundRobinGroupChat for the single-agent baseline.
MetaGPT (Hong et al., 2023)
Role-specialized agents for software engineering. We borrow the Planner–Coder–Reviewer pattern but apply it to end-to-end structured data analytics across multiple real-world domains.
StructGPT (Sui et al., 2023)
Framework for LLM reasoning over structured data via schema-aware decomposition. Highlights the challenges our ResearchAgent directly addresses through explicit methodological guidance before code generation.
G-Eval / MT-Bench (Liu et al.; Zheng et al., 2023)
LLM-as-judge evaluation with structured rubrics. We adapt this paradigm for data analysis outputs, scoring across correctness, visualization quality, methodology, and clarity.

Our work differs from prior literature by focusing specifically on end-to-end structured data analytics — where correctness depends not only on code execution but also on methodological appropriateness and visualization quality — and by providing a controlled comparative study under identical model and execution settings.

03

Method

Both pipeline modes share the same LLM backend (GPT-OSS 20B via Ollama Cloud) and code executor (AutoGen Python execution sandbox with local command-line runtime). The only variable is orchestration strategy — ensuring all differences in output quality are attributable to architecture, not model capability or tooling.

Single-Agent Baseline

A single Analyst agent handles the full workflow: inspecting the dataset, deciding an analytical strategy, generating Python code, executing it, creating visualizations, and writing the final response. Operates through AutoGen's RoundRobinGroupChat. Efficient but must internally manage every stage of reasoning and validation with no separate planning or review role.

Multi-Agent Pipeline

SelectorGroupChat routes messages using an LLM-based selector prompt. The expected order is Planner → ResearchAgent → DataScientist → CodeReviewerAgent → Result Summarizer. BLOCKING_FIXES returns control to DataScientist for revision; APPROVED advances to final summarization. The staged design creates explicit opportunities to catch incorrect assumptions, weak visualizations, and incomplete reasoning.

Why Specialized Agents?

The Planner–Researcher–Coder–Reviewer pattern mirrors how a human data analysis team would normally operate, cleanly separating concerns a single agent must implicitly juggle — reducing error propagation. The design assumption is that many failures in LLM-based analytics arise not from weak code generation alone, but because a single model is asked to simultaneously plan, reason, implement, debug, validate, and explain. Making intermediate reasoning explicit creates more opportunities to catch incorrect assumptions before they reach the user. Using categorical feedback (BLOCKING_FIXES vs APPROVED) forces the reviewer to commit to a decision, making failures diagnosable and preventing indefinite loops.

Evaluation Framework

Evaluating agentic systems is harder than evaluating static model outputs — both the final answer and the process of reaching it matter. For each benchmark task, both pipelines are executed independently and the harness records full agent conversation transcripts, generated Python code artifacts, visualization outputs, execution latency, runtime errors, and artifact manifests. For subjective and open-ended tasks, we employ an LLM-as-a-judge framework that performs structured pairwise comparison between pipeline outputs.

Ground Truth for Numerical Tasks

scripts/baseline_eval.py runs once before any pipeline evaluation to generate reference values: total rows/columns, mean discounted_price, top-5 product categories, and missing-value rates. These serve as the single source of truth for both scorers.

LLM-as-Judge for Subjective Tasks

For tasks whose output is primarily analytical prose or open-ended interpretation, numerical matching is inapplicable. The judge model receives the original task, expected output description, full execution transcripts from both pipelines, and generated visual artifacts as embedded image inputs — enabling multimodal assessment of chart quality. The judge assigns 1–5 scores across six rubric dimensions adapted from G-Eval (Liu et al., 2023):

Key Finding

Multi-agent wins 8/10 tasks on composite score. Single-agent wins only on trivially factual tasks (T1, T3) where the coordination overhead erases the margin. On hard tasks (T6, T7, T9) the multi-agent advantage is 20–30 points, driven primarily by the CodeReviewerAgent catching mislabeled visualizations, incomplete subgroup analyses, and logical inconsistencies in code-generated results.

Discussion

When Multi-Agent Wins

The multi-agent advantage is largest when: (1) the task requires multi-step reasoning or structured validation that a single internal reasoning loop handles unreliably; (2) the output is a visualization with specific formatting requirements that a one-shot pass often violates; or (3) the task involves grouped aggregation, fairness-style subgroup analysis, or robustness testing where silent logic errors are easy to introduce. The ResearchAgent's methodological guidance and CodeReviewerAgent's verification loop jointly prevent the most common failure modes of LLM-based data analysis.

When Single-Agent Is Sufficient

For simple descriptive analytics tasks — direct filtering, aggregation, schema inspection, or straightforward summary statistics — the single-agent baseline performed identically in correctness while running 3–6× faster. The coordination overhead of multiple specialized agents is not justified when task complexity is low and analytical ambiguity is limited.

Practical Recommendation

A lightweight task-complexity router that dispatches simple requests to the single-agent baseline while reserving the full collaborative pipeline for analytically demanding queries offers the best of both worlds — and is our primary future work direction.

Limitations

  • Latency overhead: sequential agent coordination and iterative reviewer loops reduce practicality for low-complexity or time-sensitive workloads.
  • Model ceiling: orchestration improves reasoning structure but cannot fully compensate for systematic deficiencies in the base model's coding or reasoning ability.
  • Evaluator subjectivity: LLM-as-a-judge scoring introduces evaluator bias despite structured rubrics and pairwise prompting.
  • Benchmark scope: the current evaluation focuses on structured tabular analytics and does not yet encompass broader machine learning workflows or multimodal tasks.

Future Work

  • Dynamic routing: classify query complexity to select single vs. multi-agent at runtime, preserving efficiency while selectively leveraging multi-agent robustness.
  • Extended workflows: expand the framework into end-to-end machine learning pipelines involving model training, evaluation, and hyperparameter tuning.
  • Multimodal verification: integrate stronger multimodal reasoning for direct visualization inspection and artifact verification to improve reviewer effectiveness.
08

Conclusion

We presented the Multi-Agent Intelligent Data Analytics Framework — a coordinated LLM-based analytics system that decomposes structured data analysis into specialized planning, research, execution, and validation stages. Implemented using Microsoft AutoGen and GPT-OSS 20B, the framework was benchmarked against a strong single-agent baseline across fifty analytical tasks spanning five real-world Kaggle datasets.

The multi-agent pipeline consistently outperforms the baseline on complex, multi-step analytical tasks requiring structured decomposition, intermediate verification, and deep reasoning, with the CodeReviewerAgent being the single highest-value component as confirmed by ablation. The single-agent baseline remains the right choice for simple descriptive queries where latency matters and additional coordination provides limited marginal benefit.

More broadly, this work demonstrates that the reliability of agentic AI systems depends not only on the strength of the underlying language model, but also on the design of the orchestration and verification mechanisms surrounding it. The key practical recommendation is a task-complexity router that directs queries to the appropriate pipeline rather than committing to one architecture universally.
