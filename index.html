Abstract
Recent advances in large language models have enabled agentic systems capable of performing structured data analysis with minimal human intervention. This paper presents a comparative study of a multi-agent orchestration system — comprising Planner, ResearchAgent, DataScientist, and CodeReviewerAgent roles built on Microsoft AutoGen AgentChat — against a single-agent baseline, evaluated across fifty analytical tasks spanning five real-world Kaggle datasets covering retail, healthcare, finance, education, and mobility domains. The multi-agent pipeline demonstrates stronger performance on analytically complex tasks requiring structured decomposition, intermediate verification, and multi-step reasoning, while the single-agent baseline remains competitive or preferable on lower-complexity descriptive analytics tasks where coordination overhead outweighs the benefit of explicit orchestration.
01

Introduction

Large language models have significantly expanded the range of tasks that can be automated through natural language interaction, including code generation, structured reasoning, and tool-assisted execution. One particularly promising application is end-to-end data analytics, where LLM-based systems can translate analytical requests into executable workflows involving data loading, cleaning, aggregation, visualization, and interpretation.

A single monolithic LLM call — the "oneshot" approach — often fails silently: it may propagate incorrect assumptions, generate misleading visualizations, or produce plausible but logically flawed conclusions without explicit failure signals. Orchestrating multiple specialized agents introduces an explicit planning-review loop that catches these errors before they reach the user.

Core Question

Does decomposing the analytics workflow into specialized collaborating agents produce measurably better outputs than a strong single-agent baseline — and for which task types is that investment worthwhile?

Contributions

  • A multi-agent data analytics framework (Planner → ResearchAgent → DataScientist → CodeReviewerAgent) built on Microsoft AutoGen AgentChat v0.7.5, using SelectorGroupChat for prompt-guided agent routing.
  • A single-agent baseline using the same underlying GPT-OSS 20B model and execution environment, operating through AutoGen's RoundRobinGroupChat, for controlled comparison.
  • A benchmark suite of 50 analytical tasks across five real-world Kaggle datasets spanning retail, healthcare, finance, education, and mobility domains.
  • Empirical results demonstrating that multi-agent orchestration consistently outperforms the baseline on complex tasks, while the single-agent baseline remains more efficient on simple descriptive tasks.
  • Evidence that verification and iterative refinement via the CodeReviewerAgent's APPROVED/BLOCKING_FIXES loop contribute substantially to output reliability.

Related Work

Our work sits at the intersection of three active research threads: LLM code generation, multi-agent coordination, and LLM-based evaluation.

AutoGen (Wu et al., 2023)
Conversational multi-agent framework. Our work builds directly on AutoGen AgentChat v0.7.5, using SelectorGroupChat for the multi-agent pipeline and RoundRobinGroupChat for the single-agent baseline.
MetaGPT (Hong et al., 2023)
Role-specialized agents for software engineering. We borrow the Planner–Coder–Reviewer pattern but apply it to end-to-end structured data analytics across multiple real-world domains.
StructGPT (Sui et al., 2023)
Framework for LLM reasoning over structured data via schema-aware decomposition. Highlights the challenges our ResearchAgent directly addresses through explicit methodological guidance before code generation begins.
G-Eval / MT-Bench (Liu et al.; Zheng et al., 2023)
LLM-as-judge evaluation with structured rubrics. We adapt this paradigm for data analysis outputs, scoring across correctness, visualization quality, methodology, and clarity in structured pairwise comparison.

Our work differs from prior literature by focusing specifically on end-to-end structured tabular data analytics — where correctness depends not only on code execution but also on methodological appropriateness and visualization quality — and by providing a controlled comparative study under identical model and execution settings.

03

Method

Both pipeline modes share the same LLM backend (GPT-OSS 20B via Ollama Cloud API) and code executor (AutoGen's LocalCommandLineCodeExecutor with a local command-line runtime). The dataset is loaded independently for each task and no persistent memory is shared across benchmark runs, preventing information leakage and ensuring a fair comparison. The only variable between pipelines is orchestration strategy.

Single-Agent Baseline

A single Analyst agent handles the full workflow: inspecting the dataset, deciding an analytical strategy, generating Python code, executing it, handling errors, creating visualizations, and writing the final response. Operates through AutoGen's RoundRobinGroupChat with only one active participant. It can iterate on its own outputs but has no separate planning, research, or review role — making it efficient but requiring the same agent to internally manage every stage of reasoning and validation.

Multi-Agent Pipeline

SelectorGroupChat routes messages using an LLM-based selector prompt. The designed execution order is Planner → ResearchAgent → DataScientist → CodeReviewerAgent. BLOCKING_FIXES returns control to DataScientist for revision; APPROVED terminates the conversation with the validated output as the final response. The conversation also terminates if the maximum message limit is reached. Missing dependencies are installed automatically before code execution.

Agent Specialization

Each agent is assigned a narrow, explicitly scoped responsibility. The Planner interprets the user request and produces an ordered analysis plan — it is explicitly not responsible for writing code, keeping planning separate from implementation. The ResearchAgent provides methodological guidance before code generation begins: identifying suitable analytical methods, recommending visualizations, and flagging risks such as misleading comparisons, data quality issues, or leakage. The DataScientist translates the plan and research guidance into executable Python code, performing dataset loading, cleaning, aggregation, visualization generation, and artifact saving, and revises its implementation when blocking issues are identified. The CodeReviewerAgent serves as the internal quality-control stage, checking execution success, logical correctness, dependency handling, and whether charts and summaries are complete and interpretable.

The main design assumption is that many failures in LLM-based analytics arise not from weak code generation alone, but because a single model is asked to simultaneously plan, reason, implement, debug, validate, and explain. Making intermediate reasoning explicit creates more opportunities to catch incorrect assumptions before they reach the user. Using categorical feedback (BLOCKING_FIXES vs APPROVED) forces the reviewer to commit to a concrete decision, making failures diagnosable and preventing indefinite loops.

Evaluation Framework

Both pipelines are executed independently through a benchmark harness for each task. The harness records full agent conversation transcripts, generated Python code artifacts, visualization outputs, execution latency, runtime errors and failure traces, and artifact manifests and metadata summaries. This logging ensures reproducibility and supports detailed post-hoc inspection of pipeline behavior.

LLM-as-a-Judge Comparative Evaluation

Because many benchmark tasks involve open-ended analytical reasoning, exact-match correctness metrics alone are insufficient. We employ an LLM-as-a-judge framework that performs structured pairwise comparison between pipeline outputs. The judge receives the original benchmark task, expected output description, optional reference guidance, full execution transcripts from both pipelines, and generated visual artifacts from both pipelines as embedded image inputs — enabling multimodal assessment of chart quality in addition to textual reasoning quality.

The judge assigns 1–5 scores across six rubric dimensions:

Overall Quality
✦
Correctness
✦

✦

✦

✦
Clarity of Explanation

The final comparative winner is selected holistically by the judge under rubric guidance rather than through a fixed weighted scoring formula. Both outputs are evaluated in the same session for a position-blind head-to-head comparison.

Efficiency Measurement

End-to-end latency is recorded for every benchmark execution using wall-clock timing measured around the complete pipeline run. This includes all LLM inference, orchestration overhead, tool execution, dependency installation, reviewer correction loops, and network latency to remote model endpoints. Judge runtime is excluded from per-pipeline latency measurements so that efficiency comparisons reflect only the execution cost of the analytical pipelines themselves.

Performance on Simple Tasks

For low-complexity descriptive analytics tasks, the single-agent baseline generally matched or exceeded the performance of the multi-agent framework. These tasks typically involved direct filtering, aggregation, schema inspection, or straightforward summary statistics — all of which required minimal decomposition and limited methodological sophistication. In these settings, the additional planning, research, and review stages introduced by the multi-agent pipeline often produced unnecessary intermediate reasoning steps without materially improving final correctness. The extra orchestration layers primarily increased latency and verbosity rather than analytical quality.

Finding — Simple Tasks

Structured multi-agent collaboration is not universally advantageous. When task complexity is low and analytical ambiguity is limited, the coordination overhead of multiple specialized agents outweighs the benefits of explicit decomposition.

Performance on Complex Tasks

The performance advantage of the multi-agent framework became substantially more pronounced on complex analytical tasks requiring multi-step reasoning, structured validation, grouped comparisons, robustness analysis, fairness-style subgroup evaluation, or protocol-driven interpretive reporting. Qualitative inspection of benchmark traces suggests that the Planner improved task decomposition by breaking compound requests into coherent analytical subtasks, while the ResearchAgent contributed methodological guidance by surfacing relevant metrics, chart recommendations, and potential analytical risks prior to implementation.

Most importantly, the reviewer loop often improved output quality by identifying issues that would otherwise have propagated into final outputs — including incomplete subgroup breakdowns, insufficient visualization labeling, omitted validation statistics, weak interpretive justification, mislabeled visualizations, and logical inconsistencies in generated code. By contrast, the single-agent baseline often produced outputs that were superficially plausible but more prone to partial reasoning, skipped validation steps, or incomplete adherence to multi-part task requirements.

The multi-agent framework improves code-generation correctness from 90% to 100% by leveraging iterative planning and reviewer-guided correction before final approval. However, this comes at a significant cost — average latency increases by approximately 3.4× and interaction count rises from 1.0 to 4.2 messages per task. These findings reinforce the broader benchmark observations: structured orchestration can substantially improve reliability, but introduces non-trivial overhead that may only be justified when analytical rigor outweighs response-time constraints.

Impact of the Validation and Review Stage

One of the most influential components of the proposed framework was the explicit validation loop between the DataScientist and CodeReviewerAgent. Across many benchmark runs, the reviewer identified blocking issues that would likely have remained uncorrected in a single-agent setting — including incorrect chart labels, omitted dependency installation, insufficient explanatory context, incomplete analytical breakdowns, and subtle logical inconsistencies in code-generated results.

This observation suggests that verification and iterative refinement may contribute as much to reliability as decomposition itself. In analytical workflows where execution success does not guarantee semantic correctness, explicit review mechanisms appear particularly valuable.

Discussion

When Multi-Agent Wins

The multi-agent advantage is largest when the task requires multi-step reasoning or structured validation that a single internal reasoning loop handles unreliably; when the output is a visualization with specific formatting requirements that a one-shot pass often violates; or when the task involves grouped aggregation, fairness-style subgroup analysis, or robustness testing where silent logic errors are easy to introduce. The ResearchAgent's methodological guidance and CodeReviewerAgent's verification loop jointly prevent the most common failure modes of LLM-based data analysis.

When Single-Agent Is Sufficient

For simple descriptive analytics tasks — direct filtering, aggregation, schema inspection, or straightforward summary statistics — the single-agent baseline performed identically in correctness while running substantially faster. The coordination overhead of multiple specialized agents is not justified when task complexity is low and analytical ambiguity is limited.

Practical Recommendation

Multi-agent orchestration should be viewed as a selective enhancement mechanism rather than a universally superior replacement for single-agent execution. A lightweight task-complexity router that dispatches simple requests to the single-agent baseline while reserving the full collaborative pipeline for analytically demanding queries offers the best of both worlds — and is our primary future work direction.

Limitations

  • Latency overhead: sequential agent coordination and iterative reviewer loops reduce practicality for low-complexity or time-sensitive workloads where benefits of deeper validation do not justify execution cost.
  • Model ceiling: orchestration improves reasoning structure and output verification but cannot fully compensate for systematic deficiencies in the base model's coding or reasoning ability.
  • Evaluator subjectivity: LLM-as-a-judge scoring introduces evaluator bias despite structured rubrics and pairwise comparative prompting, remaining an imperfect proxy for human judgment.
  • Benchmark scope: the current evaluation focuses on structured tabular analytics and does not yet encompass broader machine learning workflows, predictive modeling pipelines, or multimodal analytical tasks.

Future Work

  • Dynamic routing: a lightweight routing model could estimate task difficulty and dispatch simple requests to the single-agent baseline while reserving the full collaborative pipeline for analytically demanding queries, preserving efficiency while selectively leveraging multi-agent robustness.
  • Extended workflows: expand the framework beyond descriptive analytics into end-to-end machine learning workflows involving model training, evaluation, and hyperparameter tuning.
  • Multimodal verification: integrate stronger multimodal reasoning capabilities for direct visualization inspection and artifact verification to further improve reviewer effectiveness.
  • Chart-grounded question answering: extend the framework so that generated visualizations become first-class interactive artifacts that users can query directly for follow-up analytical insights, supporting iterative exploration over produced charts and plots.
08

Conclusion

We presented the Multi-Agent Intelligent Data Analytics Framework — a coordinated large language model based analytics system that decomposes structured data analysis into specialized planning, research, execution, and validation stages. Implemented using Microsoft AutoGen AgentChat v0.7.5 and GPT-OSS 20B, the framework was designed to evaluate whether explicit role specialization improves analytical performance relative to a strong single-agent baseline.

Through controlled benchmark evaluation across fifty tasks spanning five real-world Kaggle datasets, we found that multi-agent orchestration offers clear advantages on analytically complex workloads requiring structured decomposition, intermediate verification, and multi-step reasoning. Conversely, the single-agent baseline remained more efficient and competitive on simpler descriptive tasks where additional coordination provided limited marginal benefit.

These findings demonstrate that the value of multi-agent orchestration is fundamentally task dependent: structured collaboration improves robustness when analytical complexity exceeds what can be reliably handled by a monolithic reasoning loop, but introduces unnecessary overhead when task demands are modest. More broadly, the reliability of agentic AI systems depends not only on the strength of the underlying language model, but also on the design of the orchestration and verification mechanisms surrounding it. Carefully structured agent decomposition can improve quality and robustness, while adaptive orchestration strategies that balance analytical rigor against execution efficiency remain the key open challenge.
